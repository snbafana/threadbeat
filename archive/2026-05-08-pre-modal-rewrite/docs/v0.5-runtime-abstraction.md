# v0.5 Runtime Abstraction

v0.5 should separate durable work state from the process that executes it. The
current hosted Pi session remains the only runtime, but heartbeats should begin
flowing through task-shaped state so later sandboxed agents can be added without
rewriting the operator surface.

## Boundary

The Fastify server is the control plane. It owns SQL state, HTTP/CLI routes,
event streams, and operator policy. Runtime providers own temporary execution.

The first provider is the existing shared hosted Pi session. Future providers
can be Modal sandboxes, local-device workers, or other remote runners.

## Durable Objects

### Task

A task is bounded work that can be queued, claimed, executed, retried, audited,
and eventually assigned to a different runtime provider.

Initial fields:

- `id`
- `heartbeat_id`
- `session_id`
- `source`: `heartbeat`, `manual`, `operator`
- `status`: `queued`, `claimed`, `running`, `succeeded`, `failed`, `cancelled`
- `runtime_provider`: initially `pi-shared-session`
- `input_snapshot`
- `output`
- `error`
- `claimed_at`
- `started_at`
- `completed_at`
- `created_at`
- `updated_at`

### Task Event

Task events become the canonical log for execution state transitions. They
should mirror the current heartbeat event style while making the unit of work
explicit.

Initial event types:

- `task_queued`
- `task_claimed`
- `task_started`
- `runtime_started`
- `runtime_delta`
- `task_succeeded`
- `task_failed`
- `task_cancelled`

## Runtime Interface

The existing `RuntimeManager` already has the right shape for the first
provider. v0.5 should narrow the interface around task execution:

```ts
export interface TaskRuntime {
  start(): Promise<void>;
  stop(): Promise<void>;
  reset(): Promise<void>;
  execute(task: RuntimeTask): Promise<RuntimeTaskResult>;
  status(): RuntimeStatus;
}
```

`PiSharedSessionRuntime` can implement this by wrapping the current `run(prompt,
heartbeatId)` path. No hosted behavior should change in the first pass.

## Migration Sequence

1. Add `tasks` and `task_events` tables behind `CREATE TABLE IF NOT EXISTS`.
2. Add TypeScript row types and DB helpers for creating, claiming, completing,
   and listing tasks.
3. In `HeartbeatExecutor`, create a task for each heartbeat run while still
   writing existing `heartbeat_runs` and `heartbeat_events`.
4. Move execution status transitions onto task events, keeping heartbeat events
   as compatibility/operator summaries.
5. Once parity is proven, let the scheduler claim tasks instead of directly
   executing heartbeats.

## Non-Goals

- No Modal implementation in v0.5.
- No multi-agent scheduling in v0.5.
- No write-enabled sandbox tools in v0.5.
- No durable interactive message log unless explicitly requested.

## Acceptance Check

v0.5 is ready when the existing hosted heartbeat path behaves the same from the
CLI, while SQL can reconstruct a unit of work from task queued to task
completed without reading Railway process logs.
